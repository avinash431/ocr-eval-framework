\author{

\maketitle

\begin{abstract}
This paper presents a practical, reproducible evaluation framework for comparing optical character recognition (OCR) systems across a curated corpus of 101 enterprise-relevant documents. The evaluation covers four document categories---complex tables and forms, handwritten Devanagari character samples, Indian-language material, and mixed-content documents including receipts and mathematical formulas---and measures seven automated metrics: character error rate (CER), word error rate (WER), word accuracy, normalized edit distance, token-level F1 with precision and recall, and per-document latency. Error decomposition into substitution, insertion, and deletion rates provides diagnostic insight into failure modes.

Phase~1 of this benchmark integrates and evaluates six OCR systems: Tesseract as a local CPU baseline, Mistral~OCR as a cloud-hosted vision-language model, Docling as a structure-aware document converter, PaddleOCR as a high-performance open-source engine, Surya as a multilingual OCR toolkit, and Sarvam Vision~OCR as an India-focused document intelligence service. Full-dataset evaluation of four models on a 15-document ground-truth subset reveals meaningful performance differences. Mistral~OCR achieved 100\% success rate and the strongest overall F1 (0.9183). On forms, Surya delivered the lowest CER (0.3028) with the highest F1 (0.7710). Error decomposition reveals that insertion errors dominate across models, with Surya exhibiting the most insertion-heavy profile (73.8\% of errors). The framework is positioned as a reusable benchmarking asset with a Phase~2 roadmap covering additional vision-language models.
\end{abstract}

\section{Introduction}
%===============================================================================

Enterprise workflows that depend on extracting text and structure from scanned documents, forms, reports, receipts, and multilingual material require OCR systems that deliver reliable accuracy under diverse conditions. Typical use cases include form digitization for regulatory compliance, invoice and receipt extraction for accounts payable automation, conversion of scanned archives into searchable repositories, multilingual document processing for global operations, and structured extraction from tables and semi-structured layouts for data pipelines.

The practical challenge confronting enterprise architects is that OCR systems differ significantly across multiple dimensions that are difficult to evaluate from vendor documentation alone. Text accuracy varies substantially depending on document type, script, and image quality. Latency and throughput profiles differ between local and cloud-hosted systems in ways that affect pipeline design. Table and layout preservation capabilities range from absent to sophisticated. Multilingual support---particularly for Indian languages such as Hindi, Tamil, Telugu, and Bengali---remains unevenly developed across the OCR ecosystem~\cite{smith2007tesseract,paddleocr2020}.

This evaluation was motivated by the observation that no single publicly available benchmark adequately addresses the intersection of these concerns for an enterprise GenAI practice team evaluating OCR systems for production adoption. The framework described here is designed to produce evidence-backed recommendations grounded in reproducible measurements rather than vendor claims.

This whitepaper is intentionally framed as a Phase~1 internal benchmark. The original project plan envisioned a broader comparison across a wider multilingual dataset; the currently verified evidence base is narrower and more defensible. This paper prioritizes accuracy of claims over breadth of ambition, and all quantitative findings should be read in light of the dataset and model coverage described below.

\section{Evaluation Framework Architecture}
%===============================================================================

The benchmark is implemented as a modular Python application designed for extensibility and reproducibility. The architecture separates concerns into four layers: model wrappers, execution orchestration, metrics computation, and reporting.

\subsection{Model Integration Layer}

All OCR model wrappers inherit from a common \texttt{BaseOCRModel} abstract class and implement a single \texttt{\_ocr\_impl(image\_path)} method. Registration is automatic via a \texttt{@register\_model} decorator---adding a new model requires only creating a Python file in the \texttt{models/} directory with no manual wiring. At the time of writing, the registry contains 10 model wrappers, of which six have been fully integrated and sanity-checked for Phase~1.

Each wrapper returns a standardized \texttt{OCRResult} object containing extracted text, structured data where available, confidence scores, latency in milliseconds, estimated cost, error information, and success status. This normalization enables fair cross-model comparison regardless of the underlying API shape.

\subsection{Execution Engine}

The evaluation engine provides three execution modes: single-document mode for rapid testing and debugging, model-level mode for running one model across the entire dataset, and batch mode for running multiple models sequentially. All results are saved to timestamped directories under \texttt{results/}, ensuring successive runs do not overwrite prior artifacts. Raw OCR outputs are preserved alongside computed metrics for post-hoc failure analysis.

\subsection{Metrics Pipeline}

The metrics module applies a normalization pipeline before computing metrics, ensuring fair comparison across models that produce different output formats. Normalization consists of Unicode NFKC normalization (critical for Devanagari and accented characters), stripping of markdown and HTML formatting artifacts, and whitespace collapse to single spaces.

Seven automated measures are computed for each document where ground truth is available:

\begin{itemize}
    \item \textbf{CER and WER}: Computed using Levenshtein distance via the \texttt{rapidfuzz} library~\cite{rapidfuzz}, normalized by ground truth length.
    \item \textbf{Word Accuracy}: Defined as $\max(1 - \text{WER}, 0)$.
    \item \textbf{Normalized Edit Distance}: Provides a 0-to-1 scale normalized by the maximum of prediction and ground truth lengths.
    \item \textbf{Token F1}: Uses Counter-based bag-of-words matching with multiplicity, correctly penalizing missing or duplicated tokens. Precision and recall are reported separately.
    \item \textbf{Error Decomposition}: Character-level edit operations decomposed into substitutions, insertions, and deletions via \texttt{rapidfuzz} edit operations.
    \item \textbf{Latency}: Wall-clock time per document including API round-trip.
\end{itemize}

CER and WER values can exceed 1.0 when the prediction is substantially longer than the ground truth---a pattern observed in receipt documents where OCR systems generate text from noisy backgrounds. This behavior is reported rather than capped, as it provides diagnostic information about model failure modes.

\textbf{Tesseract}~\cite{smith2007tesseract} serves as the reference baseline. It is open-source, runs on CPU, and requires no API credentials. The Phase~1 configuration uses a multi-language pack (\texttt{eng+hin+tel+tam+ben}).

\textbf{Mistral~OCR} is accessed via the Mistral~AI cloud API and represents the vision-language model class. It achieved 100\% success rate across all 91 documents, including all 30 handwritten Devanagari character images where Tesseract failed entirely. Its output is markdown-formatted, which the normalization pipeline strips before metric computation.

\textbf{Docling} is an open-source document conversion library that prioritizes preserving document structure, producing markdown output retaining heading hierarchies and table structures. Its processing time exceeds 60 seconds per document, limiting full-dataset evaluation.

\textbf{PaddleOCR}~\cite{paddleocr2020} represents the high-performance open-source OCR category. It runs locally and delivered the lowest CER on the sanity-check sample (0.3058). Integration required significant wrapper adaptation for API version compatibility.

\textbf{Surya} is a multilingual OCR toolkit that achieved competitive accuracy (CER 0.3388, F1 0.7407) at higher latency (13,437~ms). The wrapper required complete modernization from legacy APIs to the current predictor architecture.

\textbf{Sarvam Vision~OCR} is a cloud-hosted document intelligence service focused on Indian languages. Integration was the most complex---the API uses an asynchronous five-step job workflow rather than synchronous request-response.

Four additional models (DeepSeek~OCR, Qwen-VL, OlmOCR, GOT-OCR) have registered wrappers and are planned for Phase~2 evaluation.

\section{Confidence Calibration}
%===============================================================================

Confidence calibration was evaluated for Tesseract and Surya---the two models reporting document-level confidence scores. Spearman rank correlation between confidence and F1 was computed across all ground-truth documents.

Tesseract shows strong positive correlation ($\rho=+0.83$, $p=0.0001$), indicating reliable quality gating---low-confidence outputs reliably predict poor extraction quality. Surya shows no significant correlation ($\rho=+0.08$, $p>0.05$), meaning its confidence values should not be used for routing without additional calibration.

This has architectural implications: Tesseract's confidence, despite lower overall accuracy, provides a reliable signal for human-in-the-loop pipelines.

\section{Integration and Engineering Observations}
%===============================================================================

A distinctive contribution of this evaluation is documenting the real integration complexity encountered when operationalizing OCR systems. Three of six Phase~1 models required significant wrapper rewrites due to API changes.

\textbf{API Stability.} PaddleOCR changed its constructor signature between versions. Surya completely restructured its module hierarchy in v0.17.x. Sarvam's actual API workflow bore no resemblance to the endpoint assumed by the original wrapper. This suggests enterprise teams should budget for ongoing wrapper maintenance.

\textbf{Asynchronous Cloud APIs.} Sarvam's integration revealed a five-step asynchronous job workflow (create job, upload file, start, poll, download ZIP) rather than synchronous request-response---invisible in benchmarks but critical for pipeline design.

\textbf{Batch Robustness.} Docling's batch evaluation surfaced a serialization failure where metadata resolved to a callable object, crashing JSON export. This class of bug only manifests at batch scale.

\section{Recommendations}
%===============================================================================

The following recommendations are based on Phase~1 evidence from four fully-evaluated models, supplemented by sanity-check data from Docling and PaddleOCR. All recommendations are provisional given the small ground-truth sample ($n=5$ forms, $n=10$ receipts).

\textbf{Structured Forms and Tables.} Surya achieved the lowest CER (0.3028) and highest F1 (0.7710), suggesting strongest extraction quality at the cost of higher latency (38,289~ms). Mistral~OCR provides the best combination of accuracy, reliability (100\% success), and moderate latency. Sarvam~OCR offers low latency (3,836~ms) with competitive F1 (0.7592) for latency-sensitive workloads.

\textbf{General-Purpose.} Mistral~OCR is the safest choice for heterogeneous document pipelines due to its 100\% success rate across all categories.

\textbf{Character-Level Recognition.} Vision-language architectures (Mistral: 30/30, Sarvam: 24/30) outperform traditional OCR (Tesseract: 0/30, Surya: 10/30) on minimal-context character inputs.

\textbf{Receipt Processing.} Both Tesseract (F1 0.54) and Surya (F1 0.67) exhibit insertion-dominant errors. Dedicated receipt models or field-extraction systems remain viable alternatives.

\textbf{Baseline Reference.} Tesseract remains appropriate for pipeline testing with well-calibrated confidence ($\rho=+0.83$), but should not be used as a production engine for complex documents.

%===============================================================================
\section{Limitations}
%===============================================================================

This evaluation has several important limitations. The ground-truth subset covers only 15 of 91 documents (16.5\%) from 2 of 7 subcategories. At $n=5$, no pairwise F1 differences reach statistical significance. Receipt ground truth was derived from Mistral~OCR output, creating circularity. The handwritten category uses 32$\times$32 pixel character crops, not page-scale documents. The Indian languages category contains a single document. Full-dataset results are available for only four of six models. Latency comparisons between local and cloud models are not directly comparable. Cost, privacy, and deployment complexity are discussed qualitatively only.

%===============================================================================
\section{Phase~2 Roadmap}
%===============================================================================

Phase~2 will address these limitations through several expansions. The ground-truth corpus will be expanded to at least 30 annotated documents per subcategory. The Indian languages category requires 15--20 additional documents across Hindi, Tamil, Telugu, and Bengali. Manually transcribed receipt ground truth will eliminate the Mistral circularity.

On the model side, DeepSeek~OCR, Qwen-VL~\cite{qwen}, OlmOCR, and GOT-OCR represent the emerging class of foundation-model-based OCR systems. Evaluating these alongside Phase~1 models will provide intelligence on whether general-purpose VLMs are displacing purpose-built OCR engines.

Extending the metrics framework to capture cost per document, structured extraction quality (table cell accuracy, layout fidelity), and deployment complexity scores would move toward a comprehensive decision-support tool.

%===============================================================================
\section{Conclusion}
%===============================================================================

This Phase~1 benchmark demonstrates that a practical, reproducible OCR evaluation framework can produce evidence-backed model comparisons across diverse enterprise document types. The framework's self-registering model architecture, automated metrics pipeline with text normalization and error decomposition, and batch evaluation engine provide a reusable foundation for ongoing OCR assessment.

Phase~1 evidence suggests clear stratification on structured forms ($n=5$). Surya achieved the lowest CER (0.3028) and highest F1 (0.7710), demonstrating strong character-level fidelity at high latency. Mistral~OCR provides the most robust cross-category coverage with 100\% success rate and competitive forms F1 (0.7549). Sarvam~OCR delivers competitive accuracy (F1 0.7592) at the lowest non-baseline latency (4,688~ms) but with selective category support. Tesseract serves as a baseline with well-calibrated confidence ($\rho=+0.83$) but trails on all accuracy metrics.

Error decomposition reveals insertion-dominant profiles across most models---Surya (73.8\%) and Sarvam (68.0\%)---suggesting that post-processing strategies focused on filtering excess text would yield the greatest accuracy improvement. These findings provide a defensible foundation for enterprise OCR selection decisions, with the caveat that they require refinement through expanded ground truth and Phase~2 model evaluation.

\section{Text Normalization Pipeline}

All OCR output and ground truth text is normalized before metric computation:
\begin{enumerate}
    \item \textbf{Unicode NFKC}---canonicalizes equivalent representations, critical for Devanagari.
    \item \textbf{Markdown/HTML stripping}---removes heading markers, bold/italic, table syntax, HTML tags, and entities.
    \item \textbf{Whitespace collapse}---all whitespace sequences reduced to single spaces.
\end{enumerate}

This eliminates systematic format-dependent bias from models producing markdown output (Mistral~OCR, Docling).
